# AI Product Importer — Implementační plán v1.1
**Verze:** 1.1  
**Aktualizace:** 2026-06-10  
**Změny oproti v1.0:** seřazení etap dle skutečných závislostí, explicitní go/no-go brány, revize priorit packages, přidán architectural decision log.

---

## Cíl MVP

Webová aplikace přijme produktové URL, zpracuje je dávkově a vyrobí:
- Shoptet-ready XLSX tabulku
- ZIP obrázků (WebP)
- Validační report

**Zásadní omezení MVP:** žádný přímý zápis do Shoptetu, žádná autonomní akce bez lidského schválení.

## Implementační etapy

### ETAPA 0 — Technický spike (CLI prototyp)
**Cíl:** ověřit, že pipeline funguje na 10 reálných URL bez UI.  
**Výstup:** CLI skript, interní CSV/JSON, měření kvality.

**Úkoly:**
1. Vybrat 2 dodavatelské weby pro pilotní test
2. HTTP fetch + uložení HTML a JSON-LD
3. Parsování: název, cena, SKU, EAN, obrázky
4. EAN checksum validace
5. Sharp pipeline: download → WebP + resize
6. OpenAI Structured Outputs: česká karta z faktů
7. Kategorizace (statický strom, CSV)
8. Výstup do `product.json`, `product.csv`, `validation.json`, `images/`

**CLI interface:**
```bash
pnpm import-product \
  --url "https://supplier.example/product" \
  --categories "./categories.csv" \
  --output "./result"
```

**Go/No-Go brána:**
- GO: méně než 30 % produktů vyžaduje ruční zásah při extrakci
- REWORK: kvalita OK ale rychlost/náklady mimo plán → optimalizovat
- STOP: >70 % produktů vyžaduje individuální ruční opravy → nestavět SaaS

---

### ETAPA 1 — Monorepo základ + infrastruktura
**Předpoklad:** Etapa 0 GO.

**Úkoly:**
- Monorepo: pnpm workspaces + Turborepo konfigurace
- Docker Compose: postgres, redis, minio, api, web, worker, crawler
- `packages/database`: Drizzle schéma + migrace (users, organizations, projects)
- `packages/domain`: TypeScript typy, stavy produktu, severity enum
- Základní auth (session nebo JWT, bez OAuth v MVP)
- Organizace + projekty CRUD
- Audit log tabulka

**Acceptance:**
- `docker compose up` nastartuje kompletní stack
- Uživatel se přihlásí, vytvoří projekt, nastavení se uloží
- Každá entita patří organizaci (row-level isolation)

---

### ETAPA 2 — URL ingestion + fronta
**Předpoklad:** Etapa 1 dokončena.

**Úkoly:**
- `POST /batches/:id/items` — příjem URL nebo CSV/XLSX
- SSRF ochrana (DNS, privátní IP, redirect kontrola, cloud metadata)
- Validace URL formátu + whitelist schémat (pouze HTTP/HTTPS)
- Vytvoření `source_items` + enqueue do BullMQ
- `fetch-source` worker: fetch, hash, uložení raw HTML do S3
- Stavový automat produktu (PENDING → FETCHING → ...)
- Progress endpoint (SSE nebo polling)

**Acceptance:**
- 100 URL lze vložit v jedné dávce
- Selhání jedné URL nezastaví ostatní
- Uživatel vidí live stav každé položky

---

### ETAPA 3 — Extrakce faktů
**Předpoklad:** Etapa 2 dokončena.

**Úkoly:**
- `packages/extraction`: JSON-LD parser, Open Graph, meta tagy, HTML heuristiky
- Ukládání `product_facts` (každý údaj s `source_type` + `source_url` + `confidence`)
- `extract-product` worker
- Playwright fallback worker (`browser-extract`) — izolovaný kontejner, timeout, bez secrets
- První 2–3 vendor adaptéry (pro pilotní weby z Etapy 0)

**Acceptance:**
- Pro pilotní weby vytěženy: název, cena, SKU/EAN, obrázky, varianty
- Každý údaj má zdrojový záznam
- Raw HTML je dohledatelné

---

### ETAPA 4 — Identita produktu + varianty
**Předpoklad:** Etapa 3 dokončena.

**Úkoly:**
- `packages/identity`: normalizace značky, MPN, EAN/GTIN validace (checksum + délka)
- Variantní model (`product_variants` s unikátními klíči)
- Detekce duplicit v rámci projektu
- Konfliktní pravidla → `validation_issues` (BLOCKER pro neplatný EAN, konflikt EAN)
- AI smí *navrhnout* split názvu na brand/model/variant, ne rozhodovat

**Acceptance:**
- Neplatný EAN → automaticky BLOCKED
- Konflikt EAN → NEEDS_REVIEW, ne přepsání
- Varianty mají unikátní klíče
- Duplicita se zobrazí uživateli

---

### ETAPA 5 — AI obsah + kategorizace
**Předpoklad:** Etapa 4 dokončena.

**Úkoly:**
- `packages/ai`: prompty, model gateway, schémata
- `generate-content` worker: vstup = vybraná fakta + styl projektu → Structured Output
- Výstup: `titleCs`, `shortDescriptionCs`, `longDescriptionCs`, `bulletPoints`, `usedFactIds`
- Bezpečnostní pravidla: žádný nový technický údaj bez fact ID, anti-prompt-injection instrukce
- `packages/categorization`: strom kategorií, confidence threshold (výchozí 0.80)
- `categorize-product` worker
- Uložení `usedFactIds` pro auditovatelnost

**Acceptance:**
- Každý technický údaj v textu pochází z fact záznamu
- Neznámá hodnota není domyšlena
- Kategorie pod prahem → NEEDS_REVIEW
- Prompt injection ze zdrojové stránky je ošetřen

---

### ETAPA 6 — Ceny + obrázky
**Předpoklad:** Etapa 5 dokončena.

**Úkoly:**
- `packages/pricing`: deterministický výpočet (source → exchange → markup → VAT → rounding)
- `calculate-price` worker: ukládá každý mezivýsledek, BLOCKER pokud DPH nejasné
- `packages/images`: download, MIME kontrola, Sharp pipeline (resize, WebP, hash), S3 upload
- `process-images` worker: deduplication, variantní přiřazení
- Image parametry: max 1600px delší strana, WebP q=80, bez upscalingu

**Acceptance:**
- Cena se nevypočítá bez jasného DPH stavu
- Obrázky jsou WebP, správně orientované, pod 500 KB
- Originál i odvozená verze propojeny v DB

---

### ETAPA 7 — Validace + review UI
**Předpoklad:** Etapy 4–6 dokončeny.

**Úkoly:**
- `validate-product` worker: všechna pravidla (BLOCKER / ERROR / WARNING / INFO)
- Next.js review UI:
  - Tabulka produktů (obrázek, značka, model, EAN, cena, kategorie, jistota, problémy, stav)
  - Filtry (READY / NEEDS_REVIEW / BLOCKED / no-image / EAN conflict / low-conf category)
  - Detail produktu (3-panel: zdroj | návrh | sources + issues)
  - Editace polí, změna kategorie, editace ceny
  - Schválení/zamítnutí obrázků
  - Hromadné schválení bezpečných produktů
  - `review_decisions` audit trail

**Acceptance:**
- Reviewer vidí zdrojový záznam pro každou hodnotu
- Každá ruční změna je auditována
- BLOCKER nelze obejít bez výslovného rozhodnutí

---

### ETAPA 8 — Export
**Předpoklad:** Etapa 7 dokončena.

**Úkoly:**
- `packages/export`: Shoptet XLSX mapování, CSV, ZIP generátor
- `generate-export` worker
- Výstup: `shoptet-import.xlsx`, `validation-report.csv`, `source-report.csv`, `images/`, `manifest.json`
- Blokované produkty nejsou v hlavním importu

**Acceptance:**
- Export projde testovacím importem v Shoptet sandboxu
- Počet řádků a obrázků odpovídá dávce
- manifest.json obsahuje správné počty

---

### ETAPA 9 — Pilot + měření
**Předpoklad:** Etapa 8 dokončena.

**Úkoly:**
- Zpracovat 50–100 reálných produktů od pilotního klienta
- Měřit: čas/produkt, AI náklady/produkt, počet oprav, počet blockerů
- Porovnat s ruční prací
- Zlatý dataset: 50 produktů s ručně ověřenými hodnotami pro regresní testy

**Go/No-Go brána:**
- GO: opakovatelný výsledek, known cena/produkt, pilotní klient schopen výstup použít
- REWORK: konkrétní slabina → opravit + zopakovat
- STOP: ekonomika nedává smysl ani po optimalizaci

## Klíčové zásady vývoje

1. Každý produkt je samostatná úloha — selhání nesmí blokovat dávku
2. Každé pole má `source_type` + `source_url`
3. AI nikdy není jediným zdrojem kritického údaje
4. Cena se počítá pouze deterministickým kódem
5. Nejasnost se zobrazuje, neskrývá
6. Browser (Playwright) je fallback, ne default
7. Vendor adaptéry jsou pluginy, ne podmínky v obecném parseru
8. Export musí být opakovatelný (idempotentní)
9. Všechny ruční změny se auditují
10. Agent navrhuje, pravidla rozhodují, člověk schvaluje výjimky

## Definition of Done pro MVP

MVP je hotové, když pilotní klient:
- Založí projekt a importuje svůj strom kategorií
- Vloží ≥50 URL v jedné dávce
- Systém je zpracuje dávkově bez pádu
- Fotografie jsou převedeny do WebP
- Jsou vytvořeny české návrhy produktových karet
- Konflikty jsou označeny a čekají na kontrolu
- Review UI umožní kontrolu a opravu
- Schválené produkty jsou exportovány jako Shoptet tabulka + ZIP
- Žádný kritický údaj není vygenerován bez zdrojového záznamu
- Pilotní klient výstup skutečně použije (reálný import)
